# 组织架构与Prompt关系说明

## 一、概念澄清

### 1.1 组织架构是什么？

**组织架构**是**数据结构**，表示公司的层级关系：
```
公司
  ├── 一级部门A
  │   ├── 二级部门A1
  │   └── 二级部门A2
  ├── 一级部门B
  │   └── 二级部门B1
  └── 一级部门C
```

### 1.2 组织架构信息在Prompt中的作用

**组织架构信息**应该**融入到Prompt的各个上下文中**，而不是作为独立的层级。

## 二、正确的Prompt结构

### 2.1 Prompt层级结构

```
公司信息（Company Context）
  ↓
部门信息（Department Context）- 包含组织架构信息
  ↓
岗位信息（Job Context）
  ↓
岗位画像（Job Profile）
  ↓
候选人简历（Candidate Resume）
```

### 2.2 组织架构信息如何融入

#### 1. 公司信息中
- 公司规模（影响组织复杂度）
- 组织特点（扁平化/层级化）

#### 2. 部门信息中（重点）
- **组织架构位置**：
  - 部门路径（如：公司/技术部/后端研发中心）
  - 部门层级（第几级部门）
  - 上级部门（了解部门在组织中的位置）
- **部门职责和文化**：
  - 部门职责（理解部门在组织中的作用）
  - 部门文化（理解部门的工作风格）
  - 团队规模（理解部门的规模）

#### 3. 岗位信息中
- 所属部门（关联到部门信息）
- 部门层级（从部门信息中获取）

## 三、为什么这样设计？

### 3.1 组织架构信息的作用

1. **理解岗位定位**：
   - 知道岗位在组织中的位置（哪个部门、第几级）
   - 知道部门的职责和文化，理解岗位在部门中的作用

2. **评估组织匹配**：
   - 评估候选人是否适合该部门/团队
   - 评估候选人的工作风格是否匹配部门文化

3. **识别风险点**：
   - 识别候选人可能不适合该组织层级的风险
   - 识别候选人可能不适合该部门文化的风险

### 3.2 示例

**场景**：评估一个"产品经理"岗位的候选人

**没有组织架构信息**：
- 只知道：岗位名称=产品经理，部门=技术部
- 无法理解：技术部的产品经理 vs 市场部的产品经理有什么区别

**有组织架构信息**：
- 部门路径：公司/技术部/产品研发中心
- 部门层级：第2级部门
- 上级部门：技术部（负责技术产品研发）
- 部门职责：负责技术产品的需求分析和产品设计
- 部门文化：注重技术深度、追求产品创新
- 团队规模：50人

**大模型可以理解**：
- 这是技术部的产品经理，需要懂技术
- 需要与研发团队密切协作
- 需要理解技术产品的特点
- 候选人如果有技术背景会更匹配

## 四、总结

- **组织架构**是数据结构（层级关系），不是Prompt的独立层级
- **组织架构信息**应该融入到**部门信息**中，作为部门上下文的一部分
- 通过组织架构信息，大模型可以：
  - 理解岗位在组织中的位置和作用
  - 评估组织匹配和文化匹配
  - 识别组织相关的风险点
